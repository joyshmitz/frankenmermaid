This document outlines the strategy for publishing frankenmermaid workspace crates to crates.io.

## Git Dependencies (Blockers)

### franken-kernel (BLOCKING)

**Source:** `https://github.com/Dicklesworthstone/asupersync.git`
**Used by:** fm-core (hard dependency)
**Types imported:** `Budget`, `Cx`, `DecisionId`, `NoCaps`, `PolicyId`, `SchemaVersion`, `TraceId`

**Resolution options:**

1. **Publish franken-kernel to crates.io first** (recommended)
   - The asupersync repo is owned by the same maintainer
   - Publish franken-kernel, then update fm-core to use the crates.io version
   - Pros: Clean dependency chain, no code duplication
   - Cons: Requires coordination with asupersync release schedule

2. **Vendor franken-kernel types into fm-core**
   - Copy the ~7 types directly into fm-core
   - Pros: No external dependency, publish immediately
   - Cons: Code duplication, must track upstream changes

**Decision:** Publish franken-kernel to crates.io first. The types are stable and the repo is maintained by the same team.

### fnx-* crates (NON-BLOCKING)

**Source:** `https://github.com/Dicklesworthstone/franken_networkx.git`
**Used by:** fm-layout (optional, behind `fnx-integration` feature flag)
**Crates:** fnx-runtime, fnx-classes, fnx-algorithms, fnx-views

**Resolution:** These are already optional via feature flags. Publishing with default features works fine. Users who need `fnx-integration` must use git dependencies until fnx-* is published.

## Publishing Order

Due to workspace path dependencies, crates must be published in dependency order:

```
1. fm-core           (depends on: franken-kernel from crates.io)
2. fm-parser         (depends on: fm-core)
3. fm-layout         (depends on: fm-core)
4. fm-render-svg     (depends on: fm-core, fm-layout)
5. fm-render-term    (depends on: fm-core, fm-layout)
6. fm-render-canvas  (depends on: fm-core, fm-layout)
7. fm-wasm           (depends on: fm-core, fm-parser, fm-layout, fm-render-svg, fm-render-canvas)
8. frankenmermaid-cli (depends on: all above)
```

## Pre-Publish Checklist

For each crate:

- [ ] `description` field set
- [ ] `repository` field set to `https://github.com/Dicklesworthstone/frankenmermaid`
- [ ] `homepage` field set
- [ ] `documentation` field set (optional, auto-generated by docs.rs)
- [ ] `keywords` field set (max 5)
- [ ] `categories` field set
- [ ] `readme` field set (if crate-specific README exists)
- [ ] No git dependencies in default feature set
- [ ] `cargo publish --dry-run` passes

## Publishing Commands

```bash
# Step 0: Ensure franken-kernel is published and update fm-core dependency

# Step 1: Dry-run all crates
for crate in fm-core fm-parser fm-layout fm-render-svg fm-render-term fm-render-canvas fm-wasm frankenmermaid-cli; do
    echo "=== $crate ==="
    cargo publish --dry-run -p $crate
done

# Step 2: Publish in order (wait ~60s between each for index update)
cargo publish -p fm-core
sleep 60
cargo publish -p fm-parser
sleep 60
cargo publish -p fm-layout
sleep 60
cargo publish -p fm-render-svg
cargo publish -p fm-render-term
cargo publish -p fm-render-canvas
sleep 60
cargo publish -p fm-wasm
sleep 60
cargo publish -p frankenmermaid-cli
```

## Post-Publish Actions

1. Update README.md with `cargo install frankenmermaid-cli` instructions
2. Tag release in git: `git tag v0.1.0 && git push origin v0.1.0`
3. Create GitHub release with changelog
4. Verify docs.rs builds for all crates

## Known Issues

1. **fm-cli name conflict:** The name `fm-cli` is already taken on crates.io (2 versions, 2,684 downloads). We must rename to `frankenmermaid-cli`.

2. **fm-regression-harness:** This crate is for internal testing only and should NOT be published. Add `publish = false` to its Cargo.toml.

3. **Edition 2024:** Requires Rust 1.95+. Most users should have this, but document the MSRV clearly.
